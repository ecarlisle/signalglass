# SignalGlass v0.1.0-alpha.3

This is a local-alpha milestone for SignalGlass, an observability and education tool for AI agent context, traces, provider requests/responses, token usage, and privacy boundaries.

After the merges of PR #6 (privacy redaction and token accounting) and PR #7 (documentation coherence and ingress hardening), SignalGlass can now be used locally to observe OpenAI-compatible agent/provider traffic, store traces, inspect trace reports, and understand how privacy and token behavior are handled. Dashboard views and first-class harness integrations are still evolving.

## Highlights

- **Privacy and redaction hardening** — Report-bound strings are sanitized consistently; custom `capturePolicy.redaction.secretPatterns` are respected in trace report output; stored trace events sanitize routing decisions and transformation summaries; the OpenAI adapter redacts before excerpting.
- **Token accounting corrections** — Token accounting better distinguishes prompt, completion, and total token usage; inference events are no longer double-counted into phase-based input/output totals.
- **Trace reports for stored live traces** — SQLite trace storage is opt-in; stored traces can be listed and inspected with terminal, JSON, and HTML reports.
- **OpenAI-compatible ingress hardening** — Provider config validation is stricter; credential-bearing provider URLs and sensitive configured headers are rejected; oversized request bodies stop accumulating after the limit is exceeded.
- **`provider_error` tracing** — Upstream non-2xx responses and forwarding failures emit `provider_error` trace events; upstream failure responses use normalized SignalGlass error envelopes instead of forwarding raw upstream error bodies.
- **Normalized error envelopes** — Upstream error bodies are no longer forwarded raw to the client, preventing potential secret leakage through error responses.
- **Local getting-started docs for Pi and OpenCode** — New guides explain local setup for developers dogfooding with Pi and OpenCode.
- **Documentation coherence** — README, docs, and specs are updated to match implemented behavior; `SignalGlass` casing is standardized.

## What changed

### Privacy and token accounting

- Report-bound strings (routing decisions, transformation summaries, excerpts, trace-level provider/model/agent/task fields) are sanitized consistently across terminal, JSON, HTML, and trace list reports via `sanitizeReportString` and `sanitizeReportValue`.
- Custom `capturePolicy.redaction.secretPatterns` are now passed through to all trace report sanitization paths, so org-specific secret patterns are respected in report output.
- Stored trace events sanitize `routingDecision` and `transformationSummary` before persistence, not just metadata and excerpts.
- Storage re-sanitizes and truncates redacted excerpts on write, and does not trust the incoming `payloadRef.redacted` flag as proof of safety.
- The OpenAI-compatible adapter redacts excerpts before truncation and marks adapter-created excerpts as privacy-filtered/redacted.
- Shared core redaction helpers (`redactSensitiveText`, `redactAndTruncateSensitiveText`, `isCredentialLikeText`) cover API keys, bearer tokens, auth/cookie/proxy headers, password/secret/token values, `.env`-style assignments, credential-like `storageKey` values, and custom secret patterns.
- `traceToAgentRun` prefers provider-reported `completionTokens` for output tokens and no longer treats `totalTokens` as output.
- Trace report token metrics prefer provider-reported `promptTokens` and `completionTokens`, keep `totalTokens` as inference usage, exclude inference events from phase-based input/output totals to avoid double-counting, and count `said` events as input when provider usage is not available.
- The `PREFIXED_API_KEY` constant replaces the provider-specific `OPENAI_STYLE_KEY` name to keep core provider-agnostic.
- `deriveTurnUsageMetadata` now returns a typed `TurnUsageMetadata` interface instead of `Record<string, number>`.

### Local ingress and provider handling

- Ingress supports OpenAI-compatible local traffic with configurable upstream providers.
- Provider config validation (`apps/ingress/src/config.ts`) validates provider IDs, kinds, base URLs, API key environment variable names, default models, model definitions, capabilities, and custom headers.
- Credential-bearing provider URLs are rejected.
- Sensitive configured headers such as `authorization`, `proxy-authorization`, `cookie`, `set-cookie`, and `x-api-key` are rejected.
- Secret values are not exposed in config validation error messages.
- Oversized request bodies stop accumulating after the configured byte limit is exceeded (413 on limit, 400 on invalid body).
- Upstream non-2xx responses and forwarding failures emit `provider_error` trace events with status, error type, and a sanitized error message.
- Upstream failure responses use normalized SignalGlass error envelopes (`{ error: { message, type, upstreamStatus } }`) instead of forwarding raw upstream error bodies, preventing secret leakage through error responses.
- A privacy regression test verifies that secret-like values in upstream error bodies do not leak into client responses or captured traces.

### Trace storage and reports

- SQLite trace storage is opt-in via `--storage <path>`.
- Stored traces can be listed with `signalglass traces --storage <path> list` (terminal or JSON).
- Stored traces can be inspected with `signalglass traces --storage <path> show <trace-id>` (terminal, JSON, or HTML).
- Standard mode stores sanitized metadata, routing decisions, transformation summaries, and redacted/bounded excerpts. It does not store full raw payloads by default.
- Trace list terminal output uses fixed-width column alignment and sanitizes provider/model/status/started fields.
- Trace reports include privacy disclaimers stating that token counts are approximate and that full raw payloads and secrets are not included.
- The CLI validates that the storage path exists before opening the database for read-only trace commands, and rejects unknown `--report` values with a clear error message.

### Documentation and local setup

- README is updated to reflect that ingress, storage, and trace reports now exist.
- New local setup and usage documentation covers general setup (`docs/getting-started.md`), Pi (`docs/getting-started-pi.md`), and OpenCode (`docs/getting-started-opencode.md`).
- Provider configuration docs (`docs/provider-config.md`) match the current config shape and validation behavior.
- Ingress docs (`docs/ingress.md`) describe current OpenAI-compatible local usage, storage opt-in, normalized error behavior, and CLI commands.
- Trace model, report contract, privacy, and spec docs are updated to reduce stale terminology and overclaims.
- Prose casing is normalized toward `SignalGlass`.
- Future work is explicitly labeled as future work rather than described as implemented.
- CodeRabbit docs comments are addressed: TraceEvent type coverage is expanded to include `transformation`, `tool_call`, `tool_result`; success/failure ingress behavior is clarified; standard-mode payload excerpt retention is clarified.

## Try it locally

```bash
# Install, build, test
pnpm install
pnpm build
pnpm test
```

Create a provider config file (`signalglass.config.json`):

```json
{
  "providers": [
    {
      "id": "openai",
      "label": "OpenAI",
      "kind": "openai-compatible",
      "baseUrl": "https://api.openai.com/v1",
      "apiKeyEnv": "OPENAI_API_KEY",
      "defaultModel": "gpt-4o",
      "models": [
        { "id": "gpt-4o" },
        { "id": "gpt-4o-mini" }
      ],
      "capabilities": {
        "streaming": false,
        "tools": true,
        "jsonMode": true
      }
    }
  ]
}
```

Set the upstream API key (do not put it in the config file):

```bash
export OPENAI_API_KEY=sk-...
```

Start the ingress server with SQLite trace persistence:

```bash
pnpm --filter @signalglass/cli dev -- ingress \
  --config signalglass.config.json \
  --port 8080 \
  --storage .signalglass/traces.db
```

Check the server:

```bash
curl http://localhost:8080/health
curl http://localhost:8080/v1/models
```

List stored traces:

```bash
pnpm --filter @signalglass/cli dev -- traces \
  --storage .signalglass/traces.db \
  list
```

Show a trace:

```bash
pnpm --filter @signalglass/cli dev -- traces \
  --storage .signalglass/traces.db \
  show <trace-id> \
  --report terminal
```

For Pi and OpenCode setup examples, see:

- `docs/getting-started-pi.md`
- `docs/getting-started-opencode.md`

## Known limitations

- Alpha/dev-oriented local usage. Not a production gateway.
- Dashboard Trace View / Payload View / Story View are not complete production UI features yet.
- No remote or cloud storage — traces live in a local SQLite database.
- No auth or access control.
- No encryption at rest.
- OpenAI-compatible ingress exists, but Pi and OpenCode setup is manual rather than first-class integration.
- Token counts remain approximate where tokenizer-specific counts are not available.
- Standard mode does not store full raw payloads by default; debug mode is available but opt-in.
- No CI pipeline — validation is manually run.

## Upgrade notes

- Pull latest `main` and rebuild (`pnpm install && pnpm build`).
- Recreate or check your provider config against the stricter validation rules — credential-bearing URLs and sensitive headers are now rejected.
- Existing local trace databases may not represent a stable long-term schema yet because this is alpha.
- Do not commit local `.signalglass/` databases, generated HTML reports, `.env` files, or raw payload captures.
- If you have tests that assert raw upstream error bodies in ingress responses, update them to expect the normalized SignalGlass error envelope.
